EMIC Study Randomization Algorithm

Setup

  • 12 conditions (3 stretch types × 2 dataset orderings, no repeat stretch per participant)
  • Pre-generate 100 blocks of 12, each block containing conditions 1–12 in shuffled order
  • Total: 1,200 slots
  • Each slot tracks: condition number, state (open/active/completed), participant ID, assignment time

Assignment (Normal Walk)

  1. Start at block 1, slot 1
  2. New participant arrives, gets the next open slot in the current block
  3. Slot becomes active, stamped with participant ID and time
  4. When participant completes, slot becomes completed
  5. If slot is already occupied (by a returning participant completion), skip it, move to next open slot
  6. Continue until all 12 slots in the block have been assigned or skipped

Heal Round

  1. Triggered when the last open slot in a block has been assigned (or skipped)
  2. Scan the block for any slots that are active but not completed
  3. Order them by assignment time, oldest first
  4. Next arrivals get these conditions sequentially, oldest incomplete first
  5. One pass only. When all incomplete conditions have been reassigned, move to the next block. Do not wait for heal round participants to complete.

Max Heal Passes

  1. After a heal round completes, check whether any of the heal-round participants have already dropped out
  2. If outstanding incomplete slots remain, run another heal round on the same block
  3. Repeat up to N times (configured by the "Max Heal Passes" control), or until all slots in the block are completed or reassigned
  4. This catches the cascading dropout problem: a participant assigned during a heal round may themselves drop out, leaving the block unbalanced again
  5. After all passes are exhausted (or no incomplete slots remain), move to the next block

Returning Participant (Previously Counted Out)

  1. Participant keeps their original condition number forever
  2. If they return AND complete, their completion fills an open slot matching their condition number in the current or next available block
  3. That slot is now completed and gets skipped during the normal walk
  4. If they return and do not complete, nothing happens. No slot assigned. No record. They are ignored by the system.

Completion

  1. Participant finishes their session
  2. Mark their slot as completed
  3. No further action needed

Edge Cases

A heal round participant also doesn't complete:

Their condition is not re-healed within the same block. The natural distribution across future blocks provides balance over time.

A returning participant completes but their condition is already filled in the current block:

Scan forward block by block until an open matching slot is found.

Two original participants with the same condition both get counted out and both return and complete:

Each fills the next available open slot matching their condition in separate future blocks.

All 100 blocks exhausted:

Generate more blocks. Same shuffle logic. Append to the sequence.

No incompletes at end of a block:

No heal round needed. Advance directly to next block.

Simultaneous Arrivals (Superposition Model)

  1. Each step has a configurable probability of being a batch arrival (Simult. Likelihood %). When triggered, 1–N participants arrive at once (Max Simultaneous).
  2. Snapshot: The system's current state (walking position or heal queue) is frozen at the moment the batch arrives. No state changes during assignment.
  3. Walking mode: Batch members are assigned sequentially to the next available block slots. If the block fills mid-batch, a heal round is triggered — and the walking members who just filled those slots appear as incompletes in the heal queue (they haven't resolved yet).
  4. Heal mode: Batch members cycle round-robin through the snapshotted heal queue. If 6 arrive and there are 2 holes, each hole gets 3 people. Max Heal Passes allows wrapping.
  5. Resolve: After all batch members are assigned, all fates are rolled simultaneously. Some complete, some drop. The system then reconciles: which holes got filled? Which are still open? Are more heal passes needed?
  6. This models real-world email blasts where many participants start at once — the system tries to plug holes but multiple people pile onto the same conditions before anyone has finished.
