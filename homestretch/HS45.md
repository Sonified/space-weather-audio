# HS45 — Upload Wavelet Audio to R2 & Wire into Study Flow

> **SHIP_IT.md:** Upload wavelet audio to R2 and wire into study flow
> - Store in R2: `emic-data/wavelet/{region}_{detrend}_{speed}.wav`
> - Study config maps condition → R2 URL for wavelet audio
> - Client: fetch WAV → `decodeAudioData()` → `load-audio` to wavelet worklet
> - Speed metadata (e.g. 1.25) for playhead math only — worklet plays at 1× sample rate

---

## Context

**Why pre-render?** Wavelet stretch requires a GPU CWT pass — computationally expensive and WebGPU-dependent. Paul stretch runs fine in a real-time AudioWorklet. So the study strategy is:
- **Paul stretch conditions** → real-time worklet (existing pipeline, works everywhere)
- **Wavelet conditions** → pre-rendered WAV from R2 (no GPU needed on participant's machine)

**Predecessor:** HS44 generates the WAV files locally via `stretch_test.html`. This task uploads them and wires the study to fetch + play them.

**Key insight:** The wavelet WAV is already stretched — the worklet just plays it back at 1× sample rate. Speed metadata is needed only for playhead math (mapping audio position → original time axis).

## Architecture Design

### R2 Key Scheme

```
wavelet-audio/{studySlug}/{region}_{detrend}_{speed}.wav
```

Examples:
```
wavelet-audio/emic-pilot/region1_raw_1.25x.wav
wavelet-audio/emic-pilot/region1_detrend_1.25x.wav
wavelet-audio/emic-pilot/region2_raw_1.25x.wav
wavelet-audio/emic-pilot/region2_detrend_1.25x.wav
```

Stored in `BUCKET` (same as audio-cache). ~4 MB each. Total for 2 regions × 2 detrend × 1 speed = ~16 MB.

### New Worker Route

```
GET  /api/wavelet-audio/:studySlug/:filename.wav   → serve WAV from R2
PUT  /api/wavelet-audio/:studySlug/:filename.wav    → upload WAV to R2
```

**File:** `cloudflare-worker/src/index.js` — add after the existing audio-cache block (~line 582).

```javascript
// Wavelet pre-rendered audio: /api/wavelet-audio/:studySlug/:filename.wav
const waveletAudioMatch = path.match(/^\/api\/wavelet-audio\/([^/]+)\/([^/]+\.wav)$/);
if (waveletAudioMatch) {
    const [, slug, filename] = waveletAudioMatch;
    const r2Key = `wavelet-audio/${slug}/${filename}`;

    if (request.method === 'GET') {
        const obj = await env.BUCKET.get(r2Key);
        if (!obj) return json({ error: 'Not found', key: r2Key }, 404);
        return new Response(obj.body, {
            headers: {
                'Content-Type': 'audio/wav',
                'Cache-Control': 'public, max-age=31536000',
                ...CORS_HEADERS,
            },
        });
    }

    if (request.method === 'PUT') {
        const wavData = await request.arrayBuffer();
        await env.BUCKET.put(r2Key, wavData, {
            httpMetadata: { contentType: 'audio/wav' },
        });
        return json({ ok: true, key: r2Key, size: wavData.byteLength });
    }
}
```

### Study Config Extension

Add `waveletAudioMap` to `experimentalDesign`:

```javascript
experimentalDesign: {
    // ... existing fields ...
    waveletAudioMap: {
        // key = analysis step identifier, value = R2 filename
        "region1_raw":     "region1_raw_1.25x.wav",
        "region1_detrend": "region1_detrend_1.25x.wav",
        "region2_raw":     "region2_raw_1.25x.wav",
        "region2_detrend": "region2_detrend_1.25x.wav",
    },
    waveletSpeed: 1.25,  // speed factor baked into the WAVs
}
```

This map tells the client which WAV to fetch for each analysis step when the condition's processing type is `wavelet`.

### Client Flow — Two Parallel Pipelines (study-flow.js)

**Key architectural insight:** For wavelet conditions, audio and spectrogram are *decoupled*. The pre-rendered WAV (~4 MB) is the audio source. The GOES chunks are the spectrogram source. They fetch in parallel, but the WAV fetch fires **first** — it's smaller and the audio path is the critical one (participants can't do anything without sound).

```
┌─ WAV fetch (~4 MB, single file) ──→ decode ──→ wavelet worklet    ← AUDIO READY
│
├─ GOES chunks (progressive) ───────→ stitch ──→ FFT → spectrogram  ← VISUAL FILLS IN
│                                        └────→ primeStretchProcessors (paul/resample)
```

On slow connections: audio works first, spectrogram fills in progressively behind it. On fast connections: both ready before the participant finishes reading the welcome modal. Either way — **audio is never blocked by spectrogram data**.

In the analysis step entry code (~line 2063), **before** the existing data fetch:

```javascript
if (step._assignedProcessing === 'wavelet') {
    const ed = studyConfig.experimentalDesign;
    const waveletMap = ed?.waveletAudioMap;
    if (waveletMap) {
        const regionKey = buildWaveletKey(step, ed);
        const filename = waveletMap[regionKey];
        if (filename) {
            const slug = studyConfig.slug || studyConfig.id;
            const wavUrl = `${getApiOrigin()}/api/wavelet-audio/${slug}/${filename}`;

            // Fire WAV fetch FIRST — audio is the critical path
            const waveletPromise = fetchAndDecodeWaveletAudio(wavUrl, step);

            // GOES chunks start in parallel — for spectrogram only
            // (existing triggerDataRender / loadAndFetchData path runs as normal)

            const samples = await waveletPromise;
            if (samples) {
                await loadPreRenderedWavelet(samples, ed.waveletSpeed || 1.25);
                // Don't return — let GOES pipeline continue for spectrogram
            }
        }
    }
    // Fallback: real-time GPU wavelet (existing path)
}
```

### New Functions (js/study-flow.js or js/audio-player.js)

#### `buildWaveletKey(step, experimentalDesign)`
Constructs the map key from step config:
```javascript
function buildWaveletKey(step, ed) {
    // Step has spacecraft + date range → derive region label
    // ed.detrend → raw vs detrend suffix
    const regionLabel = step.label || step.regionId || 'region1';
    const detrend = ed.detrend ? 'detrend' : 'raw';
    return `${regionLabel}_${detrend}`;
}
```

#### `fetchAndDecodeWaveletAudio(url, step)`
Fetches WAV from R2, decodes to Float32Array:
```javascript
async function fetchAndDecodeWaveletAudio(url, step) {
    const resp = await fetch(url);
    if (!resp.ok) {
        console.warn(`[WAVELET] Pre-rendered audio not found: ${url}`);
        return null;
    }
    const blob = await resp.blob();
    const { decodeWAVBlob } = await import('./data-fetcher.js');
    const result = await decodeWAVBlob(blob, {
        startTime: step.startTime,
        endTime: step.endTime,
    });
    return result.samples;
}
```

#### `loadPreRenderedWavelet(samples, bakedSpeed)`
Loads samples into the wavelet worklet directly, bypassing GPU CWT:
```javascript
async function loadPreRenderedWavelet(samples, bakedSpeed) {
    const { switchStretchAlgorithm } = await import('./audio-player.js');

    // Switch to wavelet algorithm (creates node if needed)
    switchStretchAlgorithm('wavelet');

    // Load pre-rendered samples directly into wavelet worklet
    // The worklet plays at 1× — samples are already stretched
    State.stretchNodes?.wavelet?.port.postMessage({
        type: 'load-audio',
        data: { samples, sampleRate: State.audioContext.sampleRate }
    });

    // Store baked speed for playhead math
    // (playhead needs to know original-time-to-audio-time mapping)
    State.waveletBakedSpeed = bakedSpeed;
}
```

### Playhead Math

The wavelet WAV is stretched — 1 second of original data = `bakedSpeed` seconds of audio. The playhead mapping in `audio-worklet-init.js` and `main-window-renderer.js` needs to account for this:

```javascript
// When computing original-time position from audio position:
const effectiveSpeed = State.waveletBakedSpeed || 1.0;
const originalTimePosition = audioPosition * effectiveSpeed;
```

This is the same math already used for real-time stretch — the `speed` value is just sourced from config instead of the worklet's dynamic speed parameter.

### Upload Workflow (admin-side, stretch_test.html or study-builder.html)

After HS44 renders the WAVs locally, an upload button in `study-builder.html` (Data Playback panel) lets the admin:

1. Select WAV files (file input)
2. Auto-detect filename pattern (or manually map to regions)
3. `PUT /api/wavelet-audio/{slug}/{filename}.wav` for each file

## Open Questions

1. **Region identification:** How does a study step map to a "region"? Is it the step label, a `regionId` field, or derived from the date range? The `buildWaveletKey()` function needs a reliable mapping.

2. **Multiple speeds:** Will there ever be more than one wavelet speed per study (e.g., 1.25× and 1.5×)? Current design assumes one speed stored in `waveletSpeed`. Per-condition speeds would need the map keys to include speed.

3. **Fallback behavior:** If the pre-rendered WAV is missing (404), should the client fall back to real-time GPU wavelet, or show an error? Current design falls back silently — but participants without WebGPU would get a broken experience.

4. **Auth on upload:** Should the PUT route require an admin key? Currently the audio-cache PUT is unauthenticated. For study data, admin key validation is probably wise.
