# HS25 — Auto-Trigger GPU Pyramid Computation

> **Goal:** Decouple spectrogram compute from presentation. Currently `renderProgressiveSpectrogram()` does FFT *and* display in one fused call. Refactor into two clean pipelines so computation runs as data streams in, and display is just "flip the lights on."

## The Core Insight

Two pipelines, not one:

1. **Compute Pipeline (HEAVY LIFT)** — data arrives → FFT → tiles → pyramid cache. Pure computation, no side effects on screen.
2. **Present Pipeline (QUICK DISPLAY)** — tiles exist → viewport setup → wire callbacks → `renderFrame()`. Milliseconds.

With `preRenderPyramid` enabled in study config, the compute pipeline runs as data streams in during welcome modals. By the time the participant clicks into analysis, the heavy lift is done. Display is instant.

**Fallback:** If user clicks through faster than compute finishes, the normal progressive path kicks in — zero degradation. The pre-render is purely opportunistic.

## Config

- **Study builder toggle:** "Auto-trigger GPU pyramid computation" in Data Playback & Presentation panel
- **Config key:** `config.experimentalDesign.preRenderPyramid` (boolean)
- **Study page reads:** `window.__STUDY_CONFIG.experimentalDesign.preRenderPyramid`

## Current Rendering Pipeline (The Problem)

```
study-flow.js::runAnalysis()
  ├── applyAnalysisConfig(step)           // sets spacecraft, dates, dataset
  ├── overlay fade-out (300ms)            // hides welcome modals
  ├── window.triggerDataRender()          // triggers data fetch
  │
  └─→ goes-cloudflare-fetcher.js         // streams chunks from Cloudflare R2
       ├── chunk arrives → processedChunks[]
       ├── growProgressiveBuffer()         // accumulates into Float32Array
       └── renderProgressiveSpectrogram()  // FUSED: compute + display
            │
            ├── FIRST CALL (progressiveInitDone = false):
            │   ├── initThreeScene()              // WebGPU renderer + materials
            │   ├── initPyramid(duration, sr)     // pyramid structure
            │   ├── setOnTileReady(callback)       // wires tile→render updates
            │   ├── initScrollZoom()
            │   └── progressiveInitDone = true
            │
            ├── EVERY CALL:
            │   ├── minimap FFT
            │   ├── uploadMainWaveformSamples()
            │   ├── uploadWaveformSamples() → drawWaveformFromMinMax()
            │   └── renderBaseTiles(audioData, sr, fftSize, 0)  ← THE HEAVY CALL
            │       ├── GPU compute FFT per tile (spectrogram-gpu-compute.js)
            │       ├── cascadeUpward() builds L1+ from L0 pairs
            │       └── onTileReady → renderFrame() shows tiles
            │
            └── FINAL CALL (isComplete = true):
                ├── setSuppressPyramidReady(false)
                └── dispatches 'pyramid-ready' + 'spectrogram-ready' events
```

**The problem:** `renderProgressiveSpectrogram()` does 5 jobs: scene init, FFT compute, waveform update, minimap update, display trigger. All fused. Adding a suppression gate would be duct tape on a monolith.

## Proposed Architecture: Refactored Separation

### New Structure

```
DATA ARRIVAL (background, no display)
  └─→ computePipeline(audioData, metadata)
       ├── initThreeScene()                    // idempotent, safe to call early
       ├── initPyramid(duration, sr)           // pyramid structure
       └── renderBaseTiles(audioData, sr, fft) // GPU FFT → tiles → pyramid cache
           ├── No onTileReady callback (null)
           └── setSuppressPyramidReady(true)

ANALYSIS START (instant display)
  └─→ presentPipeline()
       ├── viewport/camera setup
       ├── waveform + minimap (fast, on-demand)
       ├── setOnTileReady(renderCallback)      // wire display
       ├── setSuppressPyramidReady(false)      // unmute
       ├── renderFrame()                       // tiles already in GPU memory
       └── dispatch 'pyramid-ready'            // instant
```

## Implementation Steps

### Step 1: Extract compute pipeline from `renderProgressiveSpectrogram()`

Create `computeSpectrogramTiles(audioData, duration, sampleRate, fftSize)` in `main-window-renderer.js`:
- Calls `initThreeScene()` (idempotent)
- Calls `initPyramid()` + `renderBaseTiles()`
- Suppresses display: `setOnTileReady(null)`, `setSuppressPyramidReady(true)`
- Returns when tiles are in GPU memory
- Does NOT touch: viewport, waveform, minimap, scroll/zoom, events

### Step 2: Extract present pipeline

Create `presentPrecomputedSpectrogram()` in `main-window-renderer.js`:
- Sets up viewport/camera from current zoom state
- Wires `setOnTileReady(renderCallback)`
- Calls `setSuppressPyramidReady(false)`
- Computes waveform + minimap (fast, on-demand)
- Calls `renderFrame()` — tiles already cached
- Dispatches `'pyramid-ready'` + `'spectrogram-ready'` events

### Step 3: Slim down `renderProgressiveSpectrogram()`

Refactor the existing function to use the extracted pieces:
- If pre-computed tiles exist → call `presentPrecomputedSpectrogram()`, return
- Otherwise → run the original fused path (backward compatible, no behavior change)

This keeps the normal (non-pre-render) path working identically.

### Step 4: Hook compute pipeline to data arrival

In `goes-cloudflare-fetcher.js`, after preloaded data completes:
- Check `config.experimentalDesign.preRenderPyramid`
- If enabled, call `computeSpectrogramTiles()` with the preloaded audio data
- Fire `'prerender-complete'` event when done

Alternative hook: listen for `'data-preload-complete'` event in `study-flow.js`:
```js
window.addEventListener('data-preload-complete', (e) => {
    if (config.experimentalDesign?.preRenderPyramid) {
        computeSpectrogramTiles(e.detail.audioData, ...);
    }
});
```

### Step 5: Section-aware cache key

Each section's pre-rendered pyramid is identified by:
```
${spacecraft}:${dataset}:${startTime}:${endTime}
```

When `presentPrecomputedSpectrogram()` runs, verify the cache key matches the current analysis step. If not (e.g., wrong section's data), fall back to normal progressive path.

**Key constraint:** Only ONE pyramid at a time (shared GPU resources). Pre-render the NEXT section only. Dispose on section switch.

## Data Preloading (Already Exists)

The study config supports data preloading via `step.dataPreload`:
- `'pageLoad'` — fetch data immediately on page load (`study-flow.js` ~866)
- `'step:N'` — fetch when step N begins (`study-flow.js` ~877)

This preloads RAW DATA (Cloudflare fetch), not spectrogram tiles. The gap between "data in memory" and "tiles computed" is what this feature fills.

## Section 1 vs Section 2

Each analysis step has its own config (`step.spacecraft`, `step.startDate`, `step.endDate`). These are different datasets.

- **Before section 1:** Pre-render using section 1's preloaded data
- **Before section 2:** Pre-render using section 2's data (preloaded via `dataPreload: 'step:N'`)
- `resetProgressiveSpectrogram()` + `disposePyramid()` already clear state between sections

## Existing Infrastructure We Reuse

- `setSuppressPyramidReady(true/false)` — suppresses display events ✅
- `setOnTileReady(null)` — suppresses per-tile render callbacks ✅
- `initThreeScene()` — idempotent (guarded by `if (material) return;`) ✅
- Tile LRU cache — tiles stay in GPU memory after compute ✅
- `getStatus()` on pyramid — can check if tiles are already computed ✅
- Data preload system — already fetches raw data during modals ✅

## Performance

For a typical GOES dataset (~10Hz, 24hr = ~864K samples):
- **Current:** ~1-3 seconds of GPU FFT computation AFTER analysis starts (user sees progressive build)
- **With pre-render:** Tiles ready instantly — display is just viewport setup + `renderFrame()` (milliseconds)
- **Perceived improvement:** Spectrogram appears immediately instead of building progressively

## Notes

- Waveform + minimap are NOT pre-rendered — they're fast enough on-demand. Only the FFT pyramid is expensive.
- `initThreeScene()` needs the canvas to exist. In study mode the canvas is in the DOM from page load (hidden behind modals), so this should work. Verify.
- If the user resizes the browser during pre-render, tiles are still valid — they're resolution-independent (the pyramid handles zoom levels).
- Tiles store their own magnitude data (Uint8Array / GPU texture), not references to audio samples. Once computed, they're self-contained. Safe across section switches.
