# HS32 — Feature Viewer Session Switching Bug

> **SHIP_IT.md:** Feature viewer (feature-viewer.html): switching between analysis sections via the 1/2 buttons in the upper right is buggy. Going from section 1 → 2 works, but switching back from 2 → 1 doesn't load correctly (appears to just show "loading" and hang). Investigate the architecture for loading/unloading analysis section data in the feature viewer.

---

## Architecture Overview

### File Structure
- **`feature-viewer.html`** — Monolithic HTML page, similar to study.html but stripped down for review
- **`js/feature-viewer.js`** — Lightweight controller (~200 lines): loads study config, applies analysis step settings, triggers data fetch, handles session switching
- **`js/main.js`** — Still loaded (application layer) but feature-viewer.js overrides the flow

### URL Parameters
```
feature-viewer.html?s=<study-slug>&pid=<participant-id>&session=<1|2>
```

### Initialization Flow (`feature-viewer.js → init()`)
1. Parse URL params → set `window.__REVIEW_MODE`, `__REVIEW_PID`, `__REVIEW_SESSION`
2. `setStudyId(slug)` — configure d1-sync module
3. `fetchStudyConfig(slug)` — fetch study config from D1
4. Apply `studyConfig.appSettings` to DOM elements
5. **Force `dataRendering` to `progressive`** (override appSettings which may set `triggered`)
6. `applyAnalysisConfig(step)` — set `window.__STUDY_CONFIG` with spacecraft/dataset/times
7. `await startStreaming(null, window.__STUDY_CONFIG)` — fetch data, render spectrogram
8. Register `review-session-change` event listener

### Session Switching Flow
Button click → update `__REVIEW_SESSION` → dispatch `CustomEvent('review-session-change')` → handler calls `applyAnalysisConfig(step)` + `await startStreaming(null, window.__STUDY_CONFIG)`

### Feature Loading
Features load automatically after data fetch completes via `data-fetcher.js` → `loadRegionsAfterDataFetch()` → `loadReviewFeatures()` (in review mode), which fetches `GET /api/study/:id/participants/:pid/features`, filters by `analysis_session == __REVIEW_SESSION`.

## Root Cause Analysis

### Symptoms
- Going 1 → 2 works fine
- Going 2 → 1 shows "loading" and hangs

### Identified Issues (in order of likelihood)

#### Issue 1: No error handling on session switch
The `review-session-change` handler has NO try-catch. If `startStreaming` throws (CDAWeb API error, AudioContext issue), the status bar stays at "loading" and the error is silently swallowed.

#### Issue 2: Missing `dataSource` element → wrong data source
`feature-viewer.html` does NOT include a `#dataSource` select element. In `streaming.js` line 153-154, it defaults to `'cdaweb'` for ALL sessions, even though the study config may specify `"dataSource": "cloudflare_r2"`.

#### Issue 3: `dataRendering` not re-forced to `progressive`
In `init()`, `dataRendering` is explicitly set to `'progressive'` BEFORE the first `startStreaming`. But in the `review-session-change` handler, this is NOT done.

#### Issue 4: Race condition with concurrent `startStreaming` calls
No guard prevents a second `startStreaming` from running while the first is still in progress.

#### Issue 5: AudioContext user gesture requirement
The async chain may lose user gesture context, causing `new AudioContext()` to be created in `suspended` state.

### Most Likely Bug Mechanism
A **silent error during the second `startStreaming` call** that goes unhandled. Could come from WebGPU device loss when Three.js renderer is reused after scene disposal, or IndexedDB cache returning corrupt/stale data.

## Proposed Fix

**File: `js/feature-viewer.js`**, session change handler (line ~170):

```js
window.addEventListener('review-session-change', async (e) => {
    const newSession = e.detail.session;
    window.__REVIEW_SESSION = newSession;
    const step = findAnalysisStep(studyConfig.steps, newSession);
    if (!step) {
        console.warn(`🔎 [Review] No analysis step for session ${newSession}`);
        return;
    }
    applyAnalysisConfig(step);
    
    // Re-force progressive rendering (same as init)
    const renderSelect = document.getElementById('dataRendering');
    if (renderSelect) {
        renderSelect.value = 'progressive';
        renderSelect.dispatchEvent(new Event('change', { bubbles: true }));
    }
    
    // Error handling
    try {
        console.log(`🔎 [Review] Switching to session ${newSession}`);
        await startStreaming(null, window.__STUDY_CONFIG);
    } catch (err) {
        console.error(`🔎 [Review] Failed to load session ${newSession}:`, err);
        const statusDiv = document.getElementById('status');
        if (statusDiv) {
            statusDiv.textContent = `Error loading session ${newSession}: ${err.message}`;
            statusDiv.className = 'status error';
        }
    }
});
```

**Additionally**, add a hidden `#dataSource` element to `feature-viewer.html`:
```html
<select id="dataSource" style="display:none;">
    <option value="cloudflare">Cloudflare</option>
    <option value="cdaweb">CDAWeb</option>
</select>
```

### Debugging Strategy
Add `console.log` checkpoints in the session switch handler to identify exactly where it hangs:
1. Before `applyAnalysisConfig`
2. After `applyAnalysisConfig`, before `startStreaming`
3. Inside `startStreaming` — after cleanup, before `initAudioWorklet`
4. After `initAudioWorklet`, before data fetch
5. After data fetch, before spectrogram render
6. After spectrogram render
